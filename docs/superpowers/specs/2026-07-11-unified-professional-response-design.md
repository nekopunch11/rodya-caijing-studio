# 财经内容台统一专业响应契约设计规范

日期：2026-07-11  
状态：已获方向批准，待书面复核  
适用项目：`rodya-caijing-studio`

## 一、目标

将财经内容台从“按使用者身份决定输出强度”重构为“统一专业研究内核，按用途选择合规表达与交付形态”。个人投资者、券商投顾、客户经理和研究人员对同一问题默认获得相同的研究深度、证据标准、方法框架与方向性结论。

客户版继续专业：保留有证据支撑的研究判断、判断依据、风险边界和跟踪指标；只移除买卖指令、目标价、收益承诺、个性化资金配置、杠杆动作和未经批准的机构正式评级。

## 二、不变范围

- 保留七个业务模块及其研究方法、公式、数据源优先级和时效规则。
- 保留“一份结构化内核，多形态渲染”。
- 保留专业版默认生成 `.docx` 的规则。
- 保留客户版按需生成的规则。
- 保留卡片、文案、PPT 仅按用户意图触发的规则。
- 保留已经定稿的 docx、卡片和 PPT 视觉系统。
- 保留港股打新具体金额、融资和申购档位的高风险前置条件。

## 三、四层响应架构

### 3.1 研究层

研究层只回答“这项研究应得出什么结论”，不判断使用者身份。所有场景执行同一模块路由、取数、时效核验、研究方法、数据降级、结论—证据—证伪闭环与质量验收。

### 3.2 受众层

受众层只决定表达边界：

- **专业研究口径**：默认口径，保留完整研究判断、假设、情景、证伪链和方法细节。
- **客户合规口径**：保留研究判断及证据，删除交易决定和个性化资金动作，并补充适用边界、风险、来源、截至日和免责声明。

无法判断受众时，默认使用专业研究口径，不主动询问用户身份。

### 3.3 形态层

形态层只决定如何交付：聊天、专业/客户版 docx、卡片、配套文案、专业/客户版 PPT 或 HTML deck。不同形态不得重新研究或新增内核之外的观点。

### 3.4 行动层

行动层只在请求涉及具体买卖动作、仓位、金额、融资、杠杆或个性化申购档位时启动。普通基本面、财报、估值、排雷、产业链和 IPO 体检不因使用者身份改变研究质量，也不提前追问资产信息。

## 四、统一工作流

1. 识别研究对象与模块；只补问完成研究所必需的信息。
2. 执行统一取数、时效、证据和方法论流程。
3. 生成唯一结构化专业研究内核。
4. 从用户措辞识别受众口径；未说明时采用专业研究口径。
5. 从用户措辞识别交付形态；未说明且未要求只在聊天回答时，单模块默认生成专业版 `.docx`。
6. 如涉及具体资金或操作动作，额外执行行动安全 Gate；输入不足时只给条件性框架与补充清单。
7. 对相应产物执行研究质量、合规、视觉与文件真实性检查后交付。

## 五、结构化内核契约

结论字段统一调整为：

```yaml
结论:
  事实摘要: ""
  研究判断: ""
  判断依据: []
  适用边界: []
  关键变量: []
  反面证据: []
  行动建议: null
```

- `事实摘要`：不含判断的核心事实。
- `研究判断`：基于证据形成的方向性结论，专业版与客户版共享。
- `判断依据`：支持研究判断的核心数据和事实。
- `适用边界`：结论成立的前提、数据限制和方法局限。
- `关键变量`：后续需要跟踪的公开指标。
- `反面证据`：与主要判断相反或可能推翻判断的证据。
- `行动建议`：默认 `null`；只有通过行动安全 Gate 后才允许生成，且永不进入客户版。

其他既有内核字段保持兼容；七模块可以增加专属字段，但不得改变上述共享语义。

## 六、客户合规口径

### 6.1 允许保留

- 基本面、盈利质量、估值位置、财报变化、风险严重性和产业约束的方向性研究判断。
- 判断所依赖的关键数字、比较坐标和公开证据。
- 数据缺口、适用边界、反面证据和可能推翻结论的公开变化。
- 中性、可验证的后续跟踪指标。

例如允许写：“本期盈利质量改善，但收入恢复弱于利润表现”；“当前估值位于近五年较高分位，市场已计入较积极的增长预期”。

### 6.2 必须移除

- 买入、卖出、持有、申购、放弃等交易指令。
- 目标价、收益承诺、精确涨跌预测和确定性事件概率。
- 个性化仓位、金额、杠杆、融资和下单时间。
- “上车、打满、重仓、首选”等行动暗示。
- 未经所属机构批准却表述为机构正式评级、观点或承诺的内容。

### 6.3 渲染要求

客户版必须同时包含研究结论、判断依据、适用边界、至少一项风险或反面证据、后续观察指标、数据截至日、来源和免责声明。客户版不得只剩数据罗列，也不得新增结构化内核之外的观点。

## 七、受众与形态触发矩阵

| 用户请求 | 受众口径 | 默认交付形态 |
|---|---|---|
| “分析一下 XX” | 专业研究 | 专业版 docx |
| “简单说说 XX” | 专业研究 | 聊天摘要 |
| “做成客户版 / 转给客户” | 客户合规 | 卡片 + 文案 |
| “给客户出份报告” | 客户合规 | 客户版 docx |
| “给客户做汇报 PPT” | 客户合规 | `.pptx` |
| “做卡片 / 朋友圈图” | 沿用当前受众；未说明则客户合规 | 卡片 + 文案 |
| “出 PPT / 晨会汇报” | 沿用当前受众；未说明则专业研究 | `.pptx` |
| “只在聊天回答” | 沿用当前受众 | 完整或摘要型聊天结果 |

客户版与卡片不再互为同义词；客户版是受众口径，卡片是交付形态。

## 八、港股打新边界

港股打新的发行体检与统计研究属于统一研究层，对所有使用者采用相同专业标准。只有用户要求“打多少、借多少、选择哪一档、何时下单”等个性化动作时，才进入行动层并要求现金、融资条件、最大可承受损失、抵押品安全垫、资金时间线和流动性条件。

客户合规口径永不包含具体申购方案、金额、仓位、融资动作、下单时间或暗盘处置指令。

## 十、测试与验收

契约测试必须证明：

1. 运行规则不再把个人自用、机构内部和对客作为研究质量分流。
2. 未说明身份时可以直接执行统一专业研究。
3. 客户版保留 `研究判断`，但排除 `行动建议`。
4. 客户版必须包含证据、边界、风险、日期、来源和免责声明。
5. 具体资金、仓位、杠杆和个性化申购请求会启动行动安全 Gate。
6. 卡片、文案、docx 和 PPT 使用同一结构化内核。
7. PPT 仅在用户明确点名时生成。
8. 七个模块均遵守统一响应契约。

场景验收至少覆盖：默认个股分析、转客户、客户版报告、客户 PPT、微信卡片、普通港股 IPO 体检、具体金额申购、客户询问“能不能买”。

## 十一、完成标准

- 同一研究问题不因使用者身份得到不同质量的研究。
- 客户版保留有证据支撑的方向性判断，同时不替客户作出交易决定。
- 受众口径与交付形态在规则和文档中完全解耦。
- 七模块、共享层、入口说明和测试没有互相冲突的旧规则。
- 现有视觉模板、研究方法、数据规则和文件默认行为不回归。
- 项目源目录验证通过后再同步 Codex 运行副本；同步后不得出现重复可发现的 Skill 备份。
