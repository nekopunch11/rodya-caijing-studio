# README 与 GitHub About 重构设计

日期：2026-07-11

状态：待用户书面复核

## 一、目标与优先级

README 按以下顺位组织信息：

1. 让使用者迅速知道这个 Skill 有什么用。
2. 让使用者通过可核验的证据感知其专业性与权威性。
3. 让使用者尽快获得安装方式。
4. 完整介绍 Skill 的模块、作用、交付形态与运行方式。
5. 在不干扰产品表达的前提下，为微信公众号「风骨」引流。

首屏先讲能力与成果，不先贴“个人投资者、投顾、客户经理”等人群标签。人群说明放到 README 后半部分。

## 二、信息架构

README 采用“成果型产品页”路线，并吸收方法论白皮书的权威证据。顺序固定为：

1. 项目名、结果型主标题、简短副标题。
2. 七种核心能力与交付结果。
3. 可核验的专业与权威证据。
4. 两种安装方式。
5. 三组经审核的真实公司成果样张。
6. 七个模块的精简说明。
7. 从用户问题到交付物的工作流程。
8. 适用人群。
9. 稳定命令、更多用法、更新与架构说明。
10. 合规声明。
11. 作者与微信公众号。

更新细节和完整目录架构允许使用折叠区，避免打断主阅读路径。

首屏不出现人群标签，不用“荐股神器”“一键暴富”等营销表达，也不作速度、收益或准确率承诺。

## 四、权威证据

权威感通过真实约束和可核验事实建立，不依赖“专业、权威、研报级”等自我评价堆叠。建议章节标题为：

> 专业不是靠语气，是靠约束

证据分为五组：

- **研究方法有出处**：展示杜邦拆解、ROIC–WACC、反向 DCF、PB–ROE、M-Score、Z-Score、Wilson 置信区间等代表方法，并说明不用模糊的“AI 综合评分”代替分析。
- **数据有纪律**：说明一手披露、权威数据库和媒体信息的优先级，以及数据截至日、来源、口径、确定性和冲突处理规则。
- **判断可证伪**：核心逻辑同时包含依据、边界、反面证据和后续跟踪指标。
- **风险有安全 Gate**：区分个人研究、机构内部和客户沟通场景，对目标价、资金比例和申购方案设置更严格的前置条件。
- **输出有质量契约**：七个模块共享研究内核、合规规则和视觉规范，并有研究安全、响应契约与 PPT 视觉测试。

不堆装饰性徽章。优先使用真实样张、方法事实和测试事实证明质量。

## 五、安装区

安装紧跟权威证据，不让用户在长篇模块说明后寻找入口。只保留两种方式：

1. 将仓库地址发给 Claude 或 Codex，让 AI 自动安装。
2. 使用一行命令：

```bash
npx skills add nekopunch11/rodya-caijing-studio
```

更新与备份卫生规则继续保留，但放到后半部分或折叠区。

## 六、成果样张与审核 Gate

样张使用真实上市公司和公开数据，必须标明数据截至日，并注明“演示样张，不构成投资建议”。三组样张为：

- 宁德时代：基本面专业 DOCX。
- 招商银行：估值客户合规卡。
- 小米集团：基本面 + 估值 + 排雷组合 PPT。

审核流程：

1. 先确认样张对象、数据截至日和展示页面。
2. 在本地生成完整样张，不写入 README。
3. 单独提交截图、完整样张和配套说明供用户审核。
4. 只有用户明确批准的版本才能加入仓库和 README。
5. 未获批准的样张不得发布到 GitHub。

每组样张在 README 只展示 1–3 张最能证明能力的截图；是否提供完整脱敏样张链接，在审核时一并确定。

## 七、七模块与工作流程

七模块说明统一压缩为三个字段：

- 回答什么问题。
- 使用哪些代表性方法。
- 默认交付什么成果。

不在 README 重复完整执行规则；详细方法继续以各模块 `SKILL.md`、`references/` 和人类说明文档为准。

工作流程用一条简洁链路表达：

> 用户问题 → 自动路由 → 时效与数据核验 → 专业研究内核 → DOCX / 卡片 / PPT

## 十、边界与验收标准

- 本轮不修改 Skill 的研究方法、执行规则、触发词或默认交付契约。
- README 不虚构用户数、Stars、准确率、收益率、机构背书或测试结果。
- 所有“通过测试”类表述必须能被仓库中的实际测试支撑。

- 安装命令应在前三个主要内容区内出现。
- 公众号引流不得早于产品全貌与适用人群说明。
- 所有真实公司样张必须经过用户逐项审核后才能发布。
